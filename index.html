Summary

In this project, we will be exploring additional subdivision schemes: the Butterfly scheme, the modified Butterfly scheme, Catmull-Clark scheme, Doo-Sabin scheme, and the polar subdivision scheme, etc. We will compare the pros and cons for each of them, and discuss the tradeoffs when choosing a subdivision scheme under different scenarios.

Problem Description

Subdividing meshes is useful for rendering a more smooth, detailed surface. Subdivision methods allow us to take a coarse, simple mesh of an object and extrapolate from it to create more smooth edges. In Project 2, we implemented loop subdivision for triangle mesh surfaces. For our final project, our group will implement alternative methods of subdivision. Loop subdivision only works for triangle meshes, so meshes made from quadrilaterals require a different subdivision scheme, such as Catmull-Clark. Even for triangle meshes, different subdivision schemes have tradeoffs. Loop subdivision can yield smoother final surfaces, while Interpolating methods like the butterfly and modified butterfly method maintain key aspects of the original mesh.

The challenges of implementing different subdivision schemes lie in the details inherent to meshes: keeping track of vertices and edges within the mesh, creating new mesh elements only when necessary, and updating the mesh correctly with each subdivision. Our team plans to implement different options for subdivision (e.g. Butterfly, modified butterfly, Catmull-Clark, Doo-Sabin) to provide functionality and benefits of these different schemes.

Accomplishments

1. Infrastructure Improvements

We cloned the code base of Project 2 to get started. But soon we found that the existing code infrastructure cannot satisfy our need for multiple subdivision schemes, because it is designed to only support Loop subdivision in Project 2. It's very inefficient to switch between different schemes.

To overcome this limitation, we added a new flag -a to arguments to determine which subdivision scheme to use. For example, one can use command ./meshedit -a doo-sabin ../dae/teapot.dae to run Doo-Sabin subdivision on scene file teapot.dae.

This dramatically improved our efficiency when we work on different subdivision schemes in parallel. It also helps us have separation of concerns and make our code modular and more readable.

Progress and Updated Work Plan

We've on track to reach our original goal of implementing multiple subdivision schemes. We've completed 2 (with Doo-Sabin being buggy), and have an extensible framework to add more subdivision schemes. With our existing implemented subdivision schemes, we've already noticed some limitations on them. For example, we found out that the Doo-Sabin scheme isn't really great on handling triangle meshes as it may produce non-coplanar faces after a subdivision run. With more subdivision schemes implemented in the following week, we should be able to identify more such differences.

In the last week, we plan to fix bugs in our current implementations, and complete two more subdivision schemes (modified Butterfly and Catmull-Clark). And we will also look into non-triangle meshes to compare the tradeoffs between these subdivision schemes under different scenarios.
